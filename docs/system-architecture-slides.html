System overview

Hybrid Human-in-the-Loop IDS Dashboard

The project is built as a staged pipeline. Each stage keeps a clear job: prepare flow features, detect suspicious behaviour, combine evidence, apply feedback, then present the result in a SOC dashboard.

Stage 1 to Stage 6B

Stage 1: Dataset preparation

CSE-CIC-IDS2018 records are sampled and separated into observable flow features, ground truth labels, and a dashboard-ready labelled sample.

Why this structure matters

The detection engines should not read answer fields such as ground truth or attack type. Those labels are kept for evaluation, while the engines work from observable network-flow features.

Stage 1

Separate features from answers

Stage 1 produces aligned files with the same alert IDs, but each file has a different role. This prevents the detector from accidentally using labels as input.

Detection stagesStage 2 rules and Stage 3 ML use feature-only input.

Feature input

Used by Signature Engine and ML prediction. It contains fields such as ports, protocol, flow duration, packet counts, rates, and TCP flag counts.

Ground truth

Used only after prediction for evaluation. It maps alert IDs to raw label, mapped attack type, and benign or malicious truth.

Dashboard sample

Used for UI demonstration and explanation. It may include labels, risk scores, and readable evidence, so it is not detector input.

Stage 2 and Stage 3

Two detection views of the same flow

The signature layer is explainable and rule-based. The XGBoost layer learns statistical patterns from flow features. Their outputs are compared instead of blindly trusting one source.

Signature Engine

Checks known flow patterns such as brute-force-like login behaviour, flood-like traffic, suspicious web patterns, botnet-like behaviour, and infiltration indicators.

Rule evidence Readable rationale Stage 2

XGBoost ML Engine

Predicts an attack class from numeric flow features. Current model artifacts are a six-class prototype and do not yet include Infiltration.

Model score Feature vector Stage 3

Important boundary

Detection engines must not use `attackType`, `groundTruth`, or other answer fields during prediction. Those fields are joined only after predictions are produced.

Stage 4

Fusion turns signals into alert priority

Stage 4 joins aligned Stage 2 and Stage 3 outputs by alert ID. It decides the attack type, risk score, confidence level, and analyst review flag.

Signature outputRule hit, predicted attack type, severity, evidence.
+
ML outputPredicted class, confidence score, model evidence.
→
Fused alertFinal risk score, decision, explanation, review flag.

Decision examples

Signature + ML agreement is strongest. ML-only evidence is discounted slightly. Signature-only evidence is still useful. Disagreement remains important because it may reveal edge cases.

Evaluation is separate

Ground truth is joined after fusion, so metrics such as accuracy, macro F1, TP/TN/FP/FN, top-K precision, and review precision evaluate the result rather than influencing prediction.

Stage 5

Human feedback changes priority, not history

The original fusion score is preserved. Feedback creates a current risk score, records the reason, and applies guardrails so dangerous classes are not suppressed too far.

Confirm Threat

Increases attention for similar patterns and keeps the case review-worthy.

Expected Activity

Can reduce repeated noisy alerts when exception memory is trusted enough.

Needs Investigation

Keeps the alert under review when the analyst cannot safely mark it benign or malicious yet.

  • fusionRiskScoreStage 4 score. It remains the baseline for reporting.
  • currentRiskScoreStage 5 score after feedback and guardrails.
  • guardrailPrevents feedback from making high-risk cases look falsely safe.

Analyst workflow

From incoming flow to reviewed alert

The human analyst sees both machine evidence and feedback context before deciding whether to escalate, suppress, or keep the alert under investigation.

What changes immediately

Local current risk score, selected feedback label, session KPI cards, and queue ordering can change during the demo.

What does not change

Source JSON, trained model artifacts, Stage 1-5 pipeline outputs, and ground-truth evaluation files are not modified by the dashboard interaction.

Current prototype boundary

What this system proves, and what remains next

The current work demonstrates a complete prototype pipeline and explainable dashboard flow. It should not be described as a production IDS.

Implemented now

Dataset sampling, feature-label separation, signature rules, XGBoost prototype, fusion, feedback adjustment, replay dashboard, and evaluation summaries.

Presentation message

The key contribution is not just classification. The project shows how automated IDS evidence, human feedback, guardrails, and operational review can be combined into a dashboard that supports analyst decision-making.
