Stage 1 Data Processing Summary

Dataset:
CSE-CIC-IDS2018

Current Status:
The project has prepared the Stage 1 data processing workflow, but the full real dataset has not been processed yet.

Data Processing Purpose:
The goal is to convert CSE-CIC-IDS2018 network flow records into dashboard-style IDS alert objects.
Stage 1 now separates observable detection features, ground truth labels, and the dashboard-ready alert sample.

Planned / Prepared Processing Steps:

1. Dataset Loading
Raw CSE-CIC-IDS2018 CSV files will be loaded from the Stage 1 raw data folder or from a Colab/KaggleHub source.

2. Column Inspection
The CSV files are inspected because different files may have different columns. Some files include source and destination IP fields, while others do not.

3. Data Cleaning
Column names are cleaned, invalid values such as NaN and infinity are handled, and rows without valid labels are removed.

4. Label Checking
The dataset Label column is used to identify whether a record is benign or malicious.

5. Stratified Sampling
Benign and attack rows will be sampled separately. Attack rows are sampled by attack label so that the output is not dominated by only one attack type.

6. Label Mapping
Raw dataset labels are mapped into cleaner dashboard attack types.

Example:
Benign -> Benign
DoS Hulk / DoS GoldenEye / DoS Slowloris -> DoS
DDoS labels -> DDoS
FTP-BruteForce / SSH-Bruteforce -> Brute Force
Bot -> Botnet
Web Attack labels -> Web Attack
Infiltration -> Infiltration

7. Ground Truth Mapping
Benign labels are converted to groundTruth = benign.
All non-benign labels are converted to groundTruth = malicious.

8. Severity Mapping
Attack types are mapped to dashboard severity levels.

Example:
Benign -> Low
Brute Force -> Medium
Web Attack -> High
Botnet -> High
DoS -> High
DDoS -> High
Infiltration -> Critical

9. Risk Score Mapping
Each attack type is assigned an initial risk score.

Example:
Benign -> 25
Brute Force -> 68
Web Attack -> 76
Botnet -> 80
DoS -> 84
DDoS -> 88
Infiltration -> 92

10. Alert Conversion
Each sampled flow record is converted into a dashboard alert object following stage-1/alert-schema.md.
The converted alert object keeps selected flow-level numeric features under flowFeatures so Stage 2 and Stage 3 can use detection inputs that come from network flow behavior rather than the dataset label.

11. Feature / Label Separation
The same sampled rows are exported into three aligned files:

Observable detection features:
stage-1/data/processed/flow-feature-sample.csv

Ground truth labels for evaluation only:
stage-1/data/processed/ground-truth.json

Dashboard-ready labelled sample:
stage-1/data/processed/sample-alerts.json

Detection engines must use flow-feature-sample.csv and must not use attackType or groundTruth from sample-alerts.json as detection input.

12. Schema Validation
Converted alerts are validated to check required fields, unique IDs, valid severity, valid groundTruth, numeric port, valid risk scores, non-empty similarityKey, and flowFeatures.

13. File Export
In Colab, the temporary outputs are:
/content/flow-feature-sample.csv
/content/ground-truth.json
/content/sample-alerts.json

After review, the outputs can be copied into stage-1/data/processed/.

Important Notes:
- Large raw CSV files should not be committed to GitHub.
- The current dashboard mock data remains unchanged.
- The processed JSON is not connected to the dashboard yet.
- sample-alerts.json is retained as a dashboard-ready labelled sample / UI sample / evaluation reference, not detector input.
- No backend or database has been added.
- No machine learning model has been trained yet.
